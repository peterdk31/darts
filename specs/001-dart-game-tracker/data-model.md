# Data Model: Dart Game Tracker

**Feature**: 001-dart-game-tracker
**Date**: 2026-05-06
**Status**: Phase 1 — derived from `spec.md` Key Entities and Functional Requirements.

This document defines the persisted and in-memory entities, their fields, relationships, validation rules, and state transitions. All persisted records carry a `schemaVersion` per Constitution Principle IV. Module-owned shapes (engine state) are described abstractly here; their concrete shape is module-private — see `contracts/game-module.md`.

---

## Entity overview

```text
Session ── has-1 ──> InProgressGame? (0 or 1)
Session ── has-many ──> CompletedGameRecord  (Session History)

Game (in-progress or completed) ── refs ──> GameType (by id)
Game ── has-many ──> Team
Game ── has-ordered-list ──> ThrowRecord
Team ── has-many ──> Player (1..4)
Player ── belongs-to ──> Team (exactly one for the duration of a game)
ThrowRecord ── refs ──> Player (by playerId)
```

---

## Persisted records

The following are stored via the `src/shared/storage/` abstraction. Each is wrapped as `{ schemaVersion: number, data: T }` per R5.

### `session` (singleton)

Global UI/session-level state held across reloads.

| Field             | Type                          | Notes |
|-------------------|-------------------------------|-------|
| `schemaVersion`   | `number`                      | Owned by shell; bump on shape change. |
| `teams`           | `Team[]`                      | Last-edited team list. May be empty. |
| `inProgressGameId`| `string \| null`              | Pointer into `inProgressGame` entry. Null when no game is active. |
| `lastUpdatedAt`   | `string` (ISO 8601)           | For debugging and history sort tie-breaking. |

**Validation**:
- `teams.length` ∈ [0, 8] (a session can be empty mid-setup; the 2-team minimum is enforced at game-start time, FR-004).
- Each team's `players.length` ∈ [1, 4] (FR-002), except a fresh, never-saved team may briefly have 0 players in the editor — this is editor state, not persisted state.

---

### `inProgressGame` (singleton, optional)

Exists only when `session.inProgressGameId !== null`.

| Field             | Type                                          | Notes |
|-------------------|-----------------------------------------------|-------|
| `schemaVersion`   | `number`                                      | Owned by shell. |
| `id`              | `string` (UUID-like)                          | Stable identifier; matches `session.inProgressGameId`. |
| `gameTypeId`      | `string` (e.g. `"x01.501"`, `"cricket"`, `"around-the-clock"`, `"x01.301"`) | Resolves via the registry. |
| `resolvedSettings`| `Record<string, unknown>`                     | The chosen values for this match's settings schema (FR-006a). |
| `teams`           | `Team[]` (snapshot)                           | Frozen at game-start; not edited mid-game (FR-003 + edge case). |
| `dartsPerPlayer`  | `number`                                      | From the `GameType`'s base rules; cached for convenience. |
| `maxTeamSize`     | `number`                                      | Max players across the game's teams; used for the dart-allotment formula (FR-008). |
| `turnOrder`       | `string[]` (team IDs)                         | Frozen team rotation order (FR-017). |
| `playerRotation`  | `Record<string /*teamId*/, string[] /*playerIds in order*/>` | Frozen per-team player order (FR-010). |
| `throws`          | `ThrowRecord[]`                               | Append-only event log; the source of truth for engine state under undo/redo. |
| `redoStack`       | `ThrowRecord[]`                               | Throws that were undone and may be redone (FR-024). Cleared whenever a new throw is appended. |
| `engineState`     | `unknown` (module-owned shape)                | Cached engine state derived from `throws`. May be regenerated by replaying through `applyThrow`. |
| `engineSchemaVersion` | `number`                                  | Owned by the game module that produced `engineState`. |
| `status`          | `"in-progress"`                               | Discriminator. Becomes `"completed"` only on transition to `CompletedGameRecord`. |
| `startedAt`       | `string` (ISO 8601)                           | |

**Validation**:
- `teams.length` ≥ 2 (FR-004).
- Every team in `teams` has 1..4 players.
- `gameTypeId` MUST resolve to a registered game type at load time; if it does not, the in-progress game is unrecoverable and the shell prompts to discard.
- `throws` is append-only during normal play; undo logically pops from `throws` and pushes to `redoStack`, redo does the reverse, but each operation results in a new persisted record (no in-place mutation across renders).

**State transitions**:

```text
(none)
  │   user starts game (FR-021 cleared, all settings resolved)
  ▼
in-progress
  │   applyThrow returns effect = "gameWon"        ──► moved to history as CompletedGameRecord
  │   user starts a NEW game (FR-022a confirmed)   ──► discarded with no history entry
  │   user reloads page                             ──► restored verbatim (FR-025)
  ▼
(none)
```

### `history` (list)

Ordered list of `CompletedGameRecord` entries. Unbounded per FR-027 (no automatic eviction).

`CompletedGameRecord` fields:

| Field             | Type                          | Notes |
|-------------------|-------------------------------|-------|
| `schemaVersion`   | `number`                      | Owned by shell. |
| `id`              | `string`                      | Stable id (carried over from `inProgressGame.id`). |
| `gameTypeId`      | `string`                      | Resolved game type at completion. |
| `resolvedSettings`| `Record<string, unknown>`     | The settings that were locked in for this match (FR-006a). |
| `teams`           | `Team[]` (snapshot)           | Snapshot at completion — display-only. |
| `winnerTeamIds`   | `string[]`                    | Usually one; multiple if game rules allow co-winners (edge case). |
| `completedAt`     | `string` (ISO 8601)           | |
| `summary`         | `unknown` (module-owned)      | Optional per-game stats (e.g., final scores, closed numbers). |

**Validation**:
- `winnerTeamIds.length` ≥ 1.
- Every id in `winnerTeamIds` MUST appear in `teams`.

---

### `prefs` (singleton)

User-controlled UI preferences. Survives reloads; not tied to any specific match.

| Field             | Type                                | Notes |
|-------------------|-------------------------------------|-------|
| `schemaVersion`   | `number`                            | Owned by shell; bump on shape change. |
| `boardTheme`      | `"traditional" \| "desaturated"`    | Default `"traditional"`. Selected via the play-screen gear menu (FR-028). |

**Validation**: every field is required. Unknown values fall back to the default.

**Default record (used when no `prefs` entry exists yet)**:

```ts
{ schemaVersion: 1, data: { boardTheme: "traditional" } }
```

The shell reads `prefs` once on mount and writes back only when the user changes a preference. Quota errors on writing `prefs` are non-fatal (the user keeps their unsaved preference for the session and is shown the standard quota modal on the next storage write that targets `session` or `inProgressGame`).

## Shared in-memory / persisted-as-snapshot types

These appear inside `inProgressGame` and `CompletedGameRecord`. They are not stored as standalone top-level records.

### `Team`

| Field         | Type     | Notes |
|---------------|----------|-------|
| `id`          | `string` | UUID-like; stable for the duration of a session. |
| `displayName` | `string` | 1..40 chars; non-empty. |
| `colorId`     | `TeamColorId` | One of the 8 palette hues per FR-002b. Auto-assigned at team-creation in palette order; user-overridable via swatch picker. |
| `players`     | `Player[]` | 1..4 entries, ordered (rotation order). |

`TeamColorId = "red" | "green" | "orange" | "purple" | "teal" | "pink" | "yellow" | "cyan"` — the 8-hue palette defined in research.md R15. Each id resolves to a CSS custom property in `src/styles/tokens.css` (e.g. `--team-color-red`, `--team-color-red-on`).

### `Player`

| Field         | Type     | Notes |
|---------------|----------|-------|
| `id`          | `string` | UUID-like; stable for the duration of a match. Display names are NOT required to be unique (FR-002). |
| `displayName` | `string` | 1..30 chars; non-empty. Uniqueness NOT enforced (FR-002, FR-002a). |

### `ThrowRecord`

| Field         | Type                                                    | Notes |
|---------------|---------------------------------------------------------|-------|
| `playerId`    | `string`                                                | Refers to `Player.id` within the in-progress game's teams. |
| `teamId`      | `string`                                                | Cached for replay efficiency. |
| `segment`     | `1..20 \| "outer-bull" \| "inner-bull" \| "miss"`      | Where the dart landed. |
| `multiplier`  | `1 \| 2 \| 3`                                          | Always 1 for `"outer-bull"`, `"inner-bull"`, and `"miss"`. |
| `score`       | `number`                                                | Computed: `segment * multiplier`, or 25 / 50 / 0 for outer-bull / inner-bull / miss respectively. |
| `timestamp`   | `string` (ISO 8601)                                     | |

**Validation**:
- `segment === "miss"` ⇒ `multiplier === 1` and `score === 0`.
- `segment === "outer-bull"` ⇒ `multiplier === 1` and `score === 25`.
- `segment === "inner-bull"` ⇒ `multiplier === 1` and `score === 50`.
- `segment` is a number 1..20 ⇒ `multiplier` ∈ {1, 2, 3}, `score === segment * multiplier`.

## Game-type metadata (compile-time; not persisted)

Each registered game type contributes a `GameTypeMetadata` to the in-memory registry. This is **code**, not data — it is the manifest exported by each game module per `contracts/game-module.md`.

| Field            | Type                       | Notes |
|------------------|----------------------------|-------|
| `id`             | `string` (unique)          | e.g. `"x01.501"`, `"x01.301"`, `"cricket"`, `"around-the-clock"`. |
| `displayName`    | `string`                   | Shown on game-select. |
| `dartsPerPlayer` | `number`                   | Base value used in the team allotment formula (FR-008). |
| `settingsSchema` | `SettingDefinition[]`      | Per-match configurable options (FR-006a). |

`SettingDefinition`:

| Field         | Type                                              | Notes |
|---------------|---------------------------------------------------|-------|
| `key`         | `string`                                          | Stable across versions; used as the key in `resolvedSettings`. |
| `label`       | `string`                                          | UI label. |
| `type`        | `"toggle" \| "integer" \| "choice"`               | Types are extensible in later versions. |
| `default`     | `unknown`                                         | Type matches `type`. |
| `constraints` | type-dependent (e.g. `{min, max}` for integer)    | |

**v1 settings schemas** (per spec FR-007):
- `x01.501`: `{ doubleOut: toggle (default false), doubleIn: toggle (default false) }`
- `x01.301`: same as `x01.501`.
- `cricket`: `[]` (none).
- `around-the-clock`: `[]` (none).

## Derived rules

The following are computed from the entities above; they are not stored.

### Team-darts allotment per turn (FR-008, FR-009)

```text
teamAllotment(team)   = dartsPerPlayer × maxTeamSize
basePerPlayer(team)   = floor(teamAllotment / team.players.length)
remainder(team)       = teamAllotment − basePerPlayer × team.players.length
allotmentForPlayer(i) = basePerPlayer + (i < remainder ? 1 : 0)   // earliest players take the remainder
```

Examples (verifies spec edge cases):
- `dartsPerPlayer=3`, `maxTeamSize=2`, this team has 1 player → allotment 6, that player throws all 6.
- `dartsPerPlayer=3`, `maxTeamSize=4`, this team has 3 players → allotment 12, players throw 4-4-4.
- `dartsPerPlayer=4`, `maxTeamSize=4`, this team has 3 players → allotment 16, players throw 6-5-5.

### Throw → score (used in ThrowRecord.score validation)

```text
score(throw) =
  throw.segment === "miss"        → 0
  throw.segment === "outer-bull"  → 25
  throw.segment === "inner-bull"  → 50
  otherwise                       → throw.segment * throw.multiplier
```

## Schema versioning and migrations

- **Shell-owned records** (`session`, `inProgressGame` envelope, `history` list, `CompletedGameRecord`) carry a shell-owned `schemaVersion`. The shell ships migration functions for each bump.
- **Module-owned records** (each game's `engineState` and any module-private data) carry a module-owned `schemaVersion` distinct from the shell's. The module ships its own `migrate(prior)` per Gate I.d / IV.b.
- A loaded record whose `schemaVersion` is higher than the running code's known max version MUST be treated as unreadable; the shell discards `inProgressGame` after user confirmation and refuses to mutate `history` (read-only fallback).

## Storage namespacing

Per R5, namespaces under the storage abstraction:

| Namespace          | Cardinality | Owner | Contents |
|--------------------|-------------|-------|----------|
| `session`          | singleton   | shell | the `session` record above |
| `inProgressGame`   | singleton   | shell | the `inProgressGame` record (or absent) |
| `history`          | list        | shell | array of `CompletedGameRecord` |
| `prefs`            | singleton   | shell | the `prefs` record (board theme, etc.) |
| `game:<gameTypeId>`| per-module  | game module | module-private state, if any (most modules will keep all state inside `engineState` and not use this namespace) |

Modules MUST NOT read or write outside `game:<their own id>`.
